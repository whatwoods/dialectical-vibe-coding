📖 这是什么？
Vibe coding 是一种新兴的编程范式：开发者不再逐行书写代码，而是通过自然语言向大语言模型描述意图，由模型生成代码，开发者负责审查、验证与引导。
这种范式下，开发者与代码的关系发生了质变——你不再是直接的代码生产者，而是：
 * 🧭 方向的制定者
 * ⚖️ 结果的判断者
 * 🧱 约束体系的构建者
这引出了一系列深层的方法论问题：如何在不直接控制生产过程的条件下保证产出质量？如何将每一次实践中获得的认识固化为可靠的制度？本文尝试将毛泽东的核心方法论工具（辩证唯物主义认识论、矛盾分析法、调查研究理论、民主集中制、战略阶段论）在 vibe coding 的技术语境中重新展开。
这不是比喻集锦，这是一次严肃的方法论移植实验。

一、认识的螺旋与它的制度化：实践论、Harness 与真理标准问题
💡 核心命题
认识不是一次完成的，而是"实践→认识→再实践→再认识"的无限螺旋。检验认识之真理性的标准，只能是社会实践，而不是主观上觉得如何。
🤖 Vibe Coding 的真理标准问题
当 AI 生成了一段代码，你凭什么判断它是对的？"看起来没问题"或"跑了一下没报错"都不是真正的实践检验。真正的实践检验需要被设计、被组织、被制度化。这就是 Harness。
⚙️ Harness：实践检验的制度化
Harness（测试约束体系）是一套自动化的验证基础设施（测试用例、类型系统、lint 规则、CI 流水线等）。
 * 客观制度化： 将实践检验从肉眼审查的“主观行为”变为可执行代码的“客观制度”。
 * 知识可积累： 将发现的 AI 错误写成测试用例，使个人头脑中的经验变为团队共享的、不可遗忘的制度性知识。
 * 质变循环： 验证从手动、线性变为自动、并行，实现更高频的实践-认识循环。
🚧 Harness 先行（写第一个 Prompt 之前）
实践先于认识，且必须是有组织的实践。Harness 的搭建应当先于功能代码的生成：
 * 先写验收标准，再写 prompt。
 * 先建基础设施，再堆功能。
 * 验证工具的正确性必须由人承担。
⚠️ 两种典型错误
 * 教条主义： 试图写一个“完美的 prompt”一次性输出完整系统，认为真理可以纯粹从理论推导中获得。
 * 经验主义： 永远靠“手动跑一下看看”，不建设 harness。行动力强，但认识无法制度化和积累。
> 🎯 正确路线： 每一次 prompt 循环应当产出三样东西：功能代码、认识（新理解）、制度（Harness 中的新规则）。三者缺一，循环就是不完整的。

二、矛盾的结构与转化：矛盾论与问题的深层分解
🔍 人与 AI 的矛盾统一体
 * 同一性： 相互依存。人的自然语言需要 AI 转化为代码，AI 的生成需要人设定方向。
 * 斗争性： 根本性张力。AI 倾向于“大概率正确”，人需要“绝对确定性”。
 * 解法： Harness 是管理这个矛盾的制度性工具，保留 AI 自由度的同时约束其不确定性。
🎯 识别主要矛盾
矛盾分析法的核心是在具体情境中识别主要矛盾。
例如构建仪表盘，面临格式不统一、组件选型未定、响应式不确定、性能不明确。数据源格式不统一往往是主要矛盾，解决它，其他问题就会降级为有约束条件的工程问题。
🧩 矛盾的特殊性与 Prompt 策略
具体问题具体分析。不存在通用的“最佳 prompt 策略”。
 * 算法问题（逻辑正确性）：Prompt 需边界条件，Harness 侧重单元测试。
 * UI 问题（视觉还原度）：Prompt 需视觉参考，Harness 侧重视觉回归测试。
 * 系统集成（接口一致性）：Prompt 需界定格式，Harness 侧重契约和集成测试。
三、调查研究的层次：你到底要弄清楚什么
没有调查就没有发言权。Vibe coding 中的调查研究分为三个层次：
 * 第一层：问题域调查（这个问题的真实结构是什么？）
   * 对象： 业务逻辑、用户场景、数据流向。这是 AI 无法替代你做 的调查。
 * 第二层：工具调查（AI 的能力边界在哪里？）
   * 对象： 当前模型、上下文长度、任务类型的具体表现。应通过实验完成，并固化为 Harness 配置。
 * 第三层：生成物调查（这段代码到底做了什么？）
   * 对象： AI 生成的代码。核心逻辑和安全关键路径必须人工阅读理解；边缘代码可依赖 harness。
📋 调查的制度化工具： 项目启动清单、能力探针 (Capability Probes)、分级代码审查协议。
四、民主集中制：Harness 的政治学
Harness 在 vibe coding 中的角色，本质上就是民主集中制的制度化。
 * 🗣️ 民主的一面： AI 在 harness 界定的边界内拥有充分的生成自由（变量命名、实现细节）。过度限制就是“命令主义”。
 * 🎯 集中的一面： AI 的一切输出必须服从 harness 的检验（类型检查、测试绿灯）。这是不可协商的硬约束。
> 💡 核心洞察： 投入精力构建更强的 harness，回报是每轮 prompt 交互可以更粗放、更高效。Harness 的强度与 prompt 的简洁性成正比。
> 
群众路线的重新理解：
 * 从 AI 中来： 对 AI 生成的未经加工的意见“去粗取精、去伪存真”。
 * 到制度中去： 将认识编码为 harness 中的规则。
 * 到用户中去： 快速出原型，用真实用户验证“方向是否正确”。
五、战略阶段论：论持久战与项目演进
复杂项目必然经历若干性质不同的阶段。认清当前阶段并执行适应策略至关重要。
| 阶段 | 主要矛盾 | 战略任务 | 核心操作 | 典型错误 / 标志 |
|---|---|---|---|---|
| 1. 战略探索 | 对问题和工具的认识不足。 | 快速积累认识，而非生产功能代码。 | 用短 prompt 做大量小实验；搭建 Harness 基础骨架。 | 错误：跳过此阶段直接开发（冒险主义）。
标志：能清晰回答 AI 能做什么、不能做什么。 |
| 2. 战略相持 | 功能复杂性与 AI 单次生成能力之间的矛盾。 | 稳步推进功能开发，持续强化 Harness。 | 聚焦可验证的功能增量；Harness 与功能代码同步增长。 | 错误：反复重构（说明还在第一阶段）；Harness 停滞。 |
| 3. 战略收官 | 质量是否达标。 | 系统性质量保障。 | 端到端测试覆盖；性能基准测试；安全审计；让 AI 批量生成并自动过滤。 | 特征：依靠强约束的 Harness 释放 AI 的批量生产力。 |
六、一条贯穿性的线索：认识如何固化为制度
每一次实践-认识循环的成果，都应该被固化为制度性的约束，而不是停留在个人经验中。
 * 发现并发逻辑遗漏边界 → 转化为 lint 规则/测试模板。
 * 发现有效 prompt 结构 → 转化为团队共享 prompt 模板。
 * 发现某类代码需人工审查 → 转化为代码审查协议条款。
Vibe coding 的长期成败，不取决于单次 prompt 的巧妙程度，而取决于你是否建立了一个随实践不断进化的制度性认识体系。
七、方法论的边界
 * 适用范围： 适合中等及以上复杂度、涉及不确定性和决策权衡的 vibe coding 项目。简单确定性任务不需要此框架。
 * 原始语境： 将原著中的“对抗性”假设（敌我矛盾）在技术语境下重新阐释为“张力”。
 * 避免过度诠释： 未触及“阶级分析”等在技术语境下缺乏非牵强对应物的概念。
